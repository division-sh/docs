---
title: "Guards"
description: "Block a handler unless a condition holds, and choose what happens when it fails."

---

A guard blocks the handler unless its [CEL](/reference/expressions-cel) check passes. It is the
first decision in the handler, after the entity is acquired.

```yaml
ticket.classified:
  guard:
    id: valid_category
    check: "payload.category in ['billing', 'technical', 'account']"
    on_fail: reject
  advances_to: assigned
```

The `check` reads `entity`, `payload`, and `policy`. A guard reads the entity **before** this
handler's own writes, so it reflects what earlier handlers left, not what this one is about to
change.

## Multiple checks

When a guard has several conditions, write them as a list under `checks` instead of one long
`&&` expression. Each entry carries its own `id`:

```yaml
guard:
  checks:
    - id: in_window
      check: "entity.retry_count < policy.max_retries"
    - id: not_locked
      check: "entity.locked == false"
  on_fail: reject
```

The reason to split them is the failure report. When a guard fails, the trace and the
dead-letter record the `id` of the check that failed, so you can see *which* condition blocked
the handler instead of just learning that some combined expression came back false. This is why
real contracts use `checks` to assert identity and idempotency on a callback, one named clause at
a time.

The list is evaluated top to bottom and stops at the first failure, so every check must pass for
the guard to pass. There is one `on_fail` for the whole guard; an individual check cannot set its
own. If you supply `checks`, the singular `check` is ignored, so use one form or the other, not
both.

## on_fail

`on_fail` chooses what happens when a check fails. It defaults to `reject`.

| `on_fail` | Behavior |
|---|---|
| `reject` (default) | Stop. The event is marked rejected. |
| `discard` | Drop silently. For expected filtering you do not want recorded as a failure. |
| `kill` | Advance the entity to a terminal state. |
| `escalate:{event}` | Emit an escalation event instead of proceeding. |

`escalate` routes the failure somewhere useful instead of just stopping:

```yaml
request.received:
  guard:
    id: rate_limit
    check: "entity.request_count < policy.max_requests_per_hour"
    on_fail: escalate:rate_limit.exceeded
  advances_to: processing
```

When a guard fails, the handler stops: no state change, no data writes, no emit. Guard failures
are business logic, not transient errors, so they are not retried. See the
[execution model](/reference/execution-model) for where the guard sits in the pipeline.
