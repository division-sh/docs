title: "Build your first flow"
description: "A support-ticket flow from an empty directory to a verified, running bundle."

You are building a support-ticket system. A ticket comes in, an agent classifies it, a
system node routes it to a resolver, and it closes. The lifecycle is
`new -> assigned -> resolved -> closed`, with an escalation loop back to `assigned`. By the
end you have a contract bundle that passes the static analyzer and boots.

<Note>
  Every contract below has been checked with `swarm verify`. The bundle uses only built-in
  scalar types and a single root flow, so it stays self-contained.
</Note>

<Steps>
  <Step title="Create the package">
    ```yaml package.yaml
    name: ticket-flow
    version: 1.0.0
    description: Support ticket classification and resolution.
    platform_version: ">=1.6.0"
    flows: []
    ```
  </Step>

  <Step title="Declare the state machine">
    Declare only states the flow can actually reach. The analyzer flags any state that nothing
    advances to.

    ```yaml schema.yaml
    initial_state: new
    terminal_states: [closed]
    states: [new, assigned, resolved, closed]
    pins:
      inputs:
        events: [ticket.created]
    required_agents:
      - role: classifier-agent
        subscribes_to: [ticket.created]
        emits: [ticket.classified]
      - role: resolver-agent
        subscribes_to: [ticket.assigned]
        emits: [ticket.resolved, ticket.escalated]
    ```
  </Step>

  <Step title="Declare the entity contract">
    Fields are declared directly under the entity type name. `escalation_count` needs an
    `initial` value (or a writer), or the analyzer reports it as uncovered.

    ```yaml entities.yaml
    ticket:
      category: text
      priority: text
      resolution: text
      resolved_by: text
      escalation_count:
        type: integer
        initial: 0
    ```
  </Step>

  <Step title="Declare the events">
    Payload fields go directly under the event name; there is no `payload:` wrapper. The
    trigger is produced outside the flow, so it is marked `swarm.source: external`.

    ```yaml events.yaml
    ticket.created:
      swarm:
        source: external
      ticket_id: text
      subject: text
      body: text
    ticket.classified:
      ticket_id: text
      category: text
      priority: text
    ticket.assigned:
      ticket_id: text
      category: text
      priority: text
    ticket.resolved:
      ticket_id: text
      resolution: text
      resolved_by: text
    ticket.escalated:
      ticket_id: text
      reason: text
    ticket.resolution_confirmed:
      ticket_id: text
    ```
  </Step>

  <Step title="Declare the system node">
    The orchestrator is deterministic. The `ticket.created` handler declares
    `create_entity: true` to mint the ticket entity; a stateful input-pin handler must declare
    an acquisition mode or boot fails. Each handler commits atomically.

    ```yaml nodes.yaml
    ticket-orchestrator:
      id: ticket-orchestrator
      execution_type: system_node
      produces:
        - ticket.assigned
        - ticket.resolution_confirmed
      subscribes_to:
        - ticket.created
        - ticket.classified
        - ticket.escalated
        - ticket.resolved
        - ticket.resolution_confirmed
      event_handlers:
        ticket.created:
          create_entity: true
          advances_to: new

        ticket.classified:
          data_accumulation:
            writes:
              - source_field: category
                target_field: category
              - source_field: priority
                target_field: priority
            source_event: ticket.classified
          advances_to: assigned
          emit: ticket.assigned

        ticket.escalated:
          guard:
            id: escalation_limit
            check: "entity.escalation_count < policy.max_escalations"
            on_fail: reject
          data_accumulation:
            writes:
              - target_field: escalation_count
                expression: "entity.escalation_count + 1"
          advances_to: assigned
          emit: ticket.assigned

        ticket.resolved:
          data_accumulation:
            writes: [resolution, resolved_by]
            source_event: ticket.resolved
          advances_to: resolved
          emit: ticket.resolution_confirmed

        ticket.resolution_confirmed:
          advances_to: closed
    ```
  </Step>

  <Step title="Declare the agents">
    Both agents run in `task` mode. They communicate by emitting events: the resolver puts the
    resolution in the `ticket.resolved` payload, and the orchestrator handler writes it to the
    entity. Agents do not write entity state directly.

    ```yaml agents.yaml
    classifier-agent:
      id: classifier-agent
      role: classifier-agent
      model_tier: haiku
      subscriptions: [ticket.created]
      emit_events: [ticket.classified]
      conversation_mode: task
      max_turns_per_task: 3

    resolver-agent:
      id: resolver-agent
      role: resolver-agent
      model_tier: sonnet
      subscriptions: [ticket.assigned]
      emit_events: [ticket.resolved, ticket.escalated]
      conversation_mode: task
      max_turns_per_task: 10
    ```

    <Note>
      To give the resolver one persistent session per ticket
      (`conversation_mode: session_per_entity`, `session_scope: entity`), the agent must live in
      a child flow, not the root. Entity and flow session scope require a flow-scoped agent. See
      [Composing flows](/build/composition).
    </Note>
  </Step>

  <Step title="Set policy">
    ```yaml policy.yaml
    sla_hours: 24
    max_escalations: 3
    ```
  </Step>

  <Step title="Write the prompts">
    One markdown file per agent, in `prompts/`. Use `{{variable}}` for policy and instance
    values.

    ```markdown prompts/classifier-agent.md
    # Classifier

    You classify support tickets by category and priority.

    Read the ticket subject and body, then emit `ticket.classified` with:
    - category: one of billing, technical, account, feature
    - priority: one of critical, high, medium, low
    ```

    ```markdown prompts/resolver-agent.md
    # Resolver

    You resolve assigned tickets.

    Draft a resolution from the ticket body. If you are confident, emit `ticket.resolved` with
    the resolution and your id. If the issue needs account access or a human, emit
    `ticket.escalated` with a reason.
    ```
  </Step>

  <Step title="Verify">
    ```bash
    swarm verify --contracts ./ticket-flow
    ```

    Expect `verify ok`. The analyzer checks payload coverage, state reachability, agent
    fulfillment, entity writer coverage, handler fields, and CEL parsing. (It may also print
    informational `lint_evidence` notes, for example that a field has no internal reader; those
    are not errors.)
  </Step>
</Steps>

## How the pieces connect

- `ticket.created` arrives from outside. The orchestrator mints a ticket entity
  (`create_entity`) and the classifier agent (subscribed to the same event) classifies it.
- The classifier emits `ticket.classified`. The orchestrator writes the category and priority
  and advances to `assigned`, emitting `ticket.assigned` for the resolver.
- The resolver emits `ticket.resolved` (the orchestrator writes the resolution, advances to
  `resolved`, and emits `ticket.resolution_confirmed`, which closes the ticket) or
  `ticket.escalated` (the guard checks the escalation count, increments it, and routes back to
  `assigned`).

Every declared state is reachable, and every emitted event has a handler or a subscriber.

## Run it

Boot the runtime against the bundle and trigger a ticket, then follow the trace. See the
[Quickstart](/quickstart) for the commands.

<Note>
  Emitted payloads are producer-complete: each declared field must be supplied by the emit
  site (an agent's emit tool, or a handler's `emit.fields`). There is no implicit entity or
  trigger-payload passthrough.
</Note>
