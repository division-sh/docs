This quickstart creates a tiny flow and runs it end to end. The flow is deterministic, just
one system node and no agents, so it advances to a terminal state on its own. You will verify
it, boot a runtime, trigger it, and watch the trace.

<Note>
  Want an agent-backed flow built from scratch? Do [Build your first flow](/build/first-flow)
  instead. It walks through a support-ticket flow with classifier and resolver agents.
</Note>

## Prerequisites

- The `swarm` binary on your PATH and a clone of the repository (see [Installation](/installation)).
- Postgres 16, as a local install or a container (shown below).
- No LLM credential. This flow has no agents, so the runtime boots without one. (Flows that use
  agents need a credential; see [Installation](/installation).)

Run the commands below from the repository root, so the default platform spec resolves.

<Steps>
  <Step title="Provide Postgres">
    Any Postgres 16 works: a local install, a managed service, or a container. The binary
    connects to `127.0.0.1:5432` (database `swarm`, user `postgres`) by default and honors the
    standard `SWARM_DB_*` and `PG*` variables for anything else.

    For a local install, create the database (`createdb swarm`). Or run a throwaway container
    that matches the defaults:

    ```bash
    docker run -d --name swarm-postgres \
      -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=swarm \
      -p 5432:5432 postgres:16
    ```
  </Step>

  <Step title="Create a minimal flow">
    Four small files in a `greeting-flow/` directory. A system node creates an entity and
    advances it to a terminal state.

    ```yaml greeting-flow/package.yaml
    name: greeting-flow
    version: 1.0.0
    platform_version: ">=1.6.0"
    flows: []
    ```

    ```yaml greeting-flow/schema.yaml
    initial_state: received
    terminal_states: [done]
    states: [received, done]
    pins:
      inputs:
        events: [greeting.requested]
    ```

    ```yaml greeting-flow/events.yaml
    greeting.requested:
      swarm:
        source: external
      message: text
    ```

    ```yaml greeting-flow/nodes.yaml
    greeter:
      id: greeter
      execution_type: system_node
      subscribes_to: [greeting.requested]
      event_handlers:
        greeting.requested:
          create_entity: true
          advances_to: done
    ```

    No `agents.yaml`: this flow has no agents, just one system node, and the runtime accepts a
    bundle that omits it.
  </Step>

  <Step title="Verify the contracts">
    ```bash
    swarm verify --contracts ./greeting-flow
    ```

    Expect `verify ok`. The static analyzer runs the same checks the runtime runs at boot.
  </Step>

  <Step title="Run it">
    Set an API token, write a payload for the trigger event, then run the flow. `swarm run`
    boots a runtime in process, publishes `greeting.requested`, and streams the run's trace. It
    is the one-shot way to run a flow; for a long-running server you operate, use `swarm serve`
    (see [Running the runtime](/operate/running-the-runtime)).

    ```bash
    export SWARM_API_TOKEN=dev-token        # any value; the local runtime and CLI share it

    echo '{"message": "hello"}' > payload.json
    swarm run --contracts ./greeting-flow --event greeting.requested --payload payload.json
    ```

    The trace shows `greeting.requested` arriving, the `greeter` node creating an entity, and
    the entity advancing to `done` (a terminal state). Press Ctrl-C to stop the local runtime
    once you have seen it.
  </Step>
</Steps>

## What just happened

1. `swarm verify` checked the bundle against the platform specification.
2. `swarm run` started a runtime backed by Postgres and published your event.
3. The `greeter` node handled `greeting.requested` in one transaction: it created an entity
   and advanced it to `done`.
4. Every step was persisted, which is why the trace could stream it and a crash could replay
   it.
