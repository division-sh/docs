This quickstart runs a flow end to end with the `swarm` CLI. `swarm run` boots a runtime
in-process, publishes your trigger event, streams the run's trace, and shuts down when the
run settles — there is no long-running server to manage.

<Note>
  New to the contract model? Skim [Core concepts](/concepts/overview), or
  [build a flow](/build/first-flow). This page assumes you already have a contract bundle.
</Note>

## Prerequisites

- The `swarm` binary on your PATH — see [Installation](/installation).
- Postgres 16. The repository ships a Compose service for it.
- A contract bundle (a directory with a `package.yaml` at its root).
- Optional: an LLM credential. The default runtime mode is scripted (`cli_test`), so a first
  run needs no provider key.

<Steps>
  <Step title="Start Postgres">
    ```bash
    docker compose up -d postgres
    ```

    Any reachable Postgres 16 works; connection settings come from the repository's `.env`.
  </Step>
  <Step title="Validate your contracts">
    ```bash
    swarm verify --contracts ./my-flow
    ```

    `swarm verify` runs the static analyzer against files on disk — no runtime needed. Fix
    any error before running; a bundle that runs has passed this check.
  </Step>
  <Step title="Run a flow">
    Write a JSON payload for the trigger event, then run it:

    ```bash
    echo '{"order_id":"o-123","priority":"high"}' > payload.json

    swarm run --contracts ./my-flow --event order.created --payload payload.json
    ```

    `swarm run` starts a runtime in-process, publishes `order.created`, and streams the run's
    trace — events, deliveries, state changes, and agent turns — until the run settles.
  </Step>
</Steps>

<Tip>
  Run from the repository root so `swarm run` picks up `.env` (database settings) and the
  default platform spec automatically. To run against an already-running runtime instead,
  start [`swarm serve`](/installation) and use `swarm run --connect <url> …`.
</Tip>

## What just happened

1. `swarm verify` checked the bundle against the platform specification.
2. `swarm run` started a runtime in-process, backed by Postgres.
3. `order.created` was validated against its payload schema, persisted, and routed to its
   subscribers by their declared subscriptions.
4. A system node handled it in one atomic transaction — advancing state, writing fields, and
   emitting follow-up events; agents reasoned and emitted events of their own.
5. The trace streamed to your terminal, and the local runtime shut down when the run settled.
