This quickstart boots a Swarm runtime against a contract bundle and triggers a run. It uses
the bundled Docker Compose setup, which runs in scripted-LLM mode by default — so a first
run works without a real provider key.

<Note>
  New to the contract model? Skim [Core concepts](/concepts/overview) first, or jump
  straight to [Build a flow](/build/first-flow) to author a bundle from scratch.
</Note>

## Prerequisites

- Docker and Docker Compose.
- A contract bundle on disk (a directory with a `package.yaml` at its root).
- Optional: an LLM provider credential. The bundled Compose file sets
  `SWARM_LLM_RUNTIME_MODE: cli_test` (a scripted runtime, no API bill). To run real agents,
  set `CLAUDE_CODE_OAUTH_TOKEN` or `ANTHROPIC_API_KEY` and switch the runtime mode to `api`.

<Steps>
  <Step title="Clone and point at a contract bundle">
    ```bash
    git clone https://github.com/division-sh/swarm && cd swarm
    export SWARM_CONTRACTS_HOST_DIR=/absolute/path/to/your/contracts
    ```

    The Compose file mounts that directory read-only at `/opt/swarm/contracts`. To run real
    agents, add a credential: `echo 'CLAUDE_CODE_OAUTH_TOKEN=...' >> .env`.
  </Step>
  <Step title="Boot the runtime">
    ```bash
    docker compose up -d postgres orchestrator
    ```

    Postgres comes up first; the orchestrator then loads your bundle, runs it through the
    static analyzer, and starts the event loop.
  </Step>
  <Step title="Confirm the runtime is up">
    ```bash
    curl -fsS http://localhost:8070/healthz
    ```

    A successful boot means your contracts passed the static analyzer. If the bundle has a
    contract error, boot aborts — see [Testing and verification](/build/testing).
  </Step>
  <Step title="Trigger a run">
    Start a run from a triggering event. `swarm run` publishes the event, allocates a run,
    and streams the run's trace — events, deliveries, state changes, and agent turns — until
    it settles.

    ```bash
    docker compose exec orchestrator swarm run \
      --event order.created \
      --payload-json '{"order_id":"o-123","priority":"high"}'
    ```

    To re-attach to a run later, use `swarm trace <run-id> -f`.
  </Step>
</Steps>

<Note>
  **Ports.** The bundled Compose file serves the runtime API on `:8070`; the standalone
  `swarm` binary defaults to `http://127.0.0.1:8081`. See [Installation](/installation) for
  the binary path.
</Note>

## What just happened

1. The orchestrator loaded your contract bundle and validated it against the platform
   specification before accepting any events.
2. `order.created` was validated against its payload schema, persisted to the event store,
   and routed to its subscribers by their declared subscriptions.
3. A system node handled it inside a single transaction — advancing state, writing fields,
   and emitting follow-up events atomically.
4. Every step was persisted, which is why `swarm trace` can reconstruct the run.
