Swarm makes a few opinionated choices and holds them. Those choices buy reproducibility,
replay, and audit. They cost YAML and a static-analyzer pass before the runtime starts.
That tradeoff is right for some workloads and wrong for others.

## The problem with LLM-managed orchestration

Most multi-agent frameworks make the LLM *be* the system: it decides who acts next, tracks the
state, and is trusted to follow the rules. LLMs are not reliable at that job, and it shows in
production:

- **The model fakes the state.** An agent reports a step done when it is not, or contradicts
  an earlier decision. Nothing catches it, because the model is also the bookkeeper: there is
  no independent source of truth.
- **Non-deterministic routing.** When an LLM decides who acts next, two runs of the same input
  diverge. You cannot reliably debug, audit, or reproduce a run.
- **Context-window exhaustion.** Put ten agents in one shared chat and the model loses the
  thread well before the tenth; quality falls as the task grows.
- **No crash recovery.** A process dies mid-turn and the in-memory state goes with it.
- **No composition.** Reusing a pipeline in another project means refactoring code, not
  importing a package.

The fix is to stop asking the LLM to be the system. Swarm splits the two jobs: **LLMs reason,
in small targeted tasks; deterministic system nodes own state, routing, and rules.** An agent
emits an event with its result, and a system node decides what is written and whether anything
advances. Routing is [derived from declared subscriptions](/concepts/events-and-routing), each
transition commits in [one transaction](/concepts/system-nodes-and-handlers), and every event
and state change is persisted, so runs are reproducible, auditable, and crash-safe.

## Strong fit

- **Long-running workflows, hours to days.** Durable timers, persistence, and crash recovery
  let a run span days without losing state; replay answers "why did the system do that, and
  when?" later.
- **Coordinating many agents, from a handful to a hundred or more.** Hierarchical addressing
  (coordinator → managers → workers), per-flow [agent sharding](/concepts/agents-and-sessions),
  and dynamic flow instances
  scale the agent count on a single deployment, without the context-window blowup of a flat
  chat.
- **Systems assembled from reusable parts.** A flow is a self-contained package with typed
  input and output pins. Build a flow once and import it into other projects, or swap one for
  a better version, without refactoring.
- **Crash recovery as a requirement.** Atomic transitions and event replay are the
  baseline, not a bolt-on.
- **Regulated or audited environments.** Two-layer event-plus-mutation persistence makes
  "what changed, when, and because of which event" a single query.
- **High-volume parallel entities.** Orders, tickets, claims: hundreds flow through one
  contract without interfering, each in its own scoped state.
- **Cost-controlled deployments.** Token usage is tracked per entity and per actor, with
  thresholds that escalate into throttle and emergency states.

## Weak fit

- **One-shot LLM calls.** Use the provider SDK directly: no runtime needed.
- **Conversational chatbots.** Short interactions where the value is the conversation; the
  contract overhead does not pay for itself.
- **Exploratory prototyping that changes hourly.** The analyzer refusing to boot a
  half-finished contract is exactly the wrong friction while you are still sketching.
- **LLM-managed routing at runtime.** Swarm refuses this by design. If "the model decides
  what happens next" is the point of your system, this is the wrong tool.

## Swarm vs. alternatives

Swarm sits closer to durable-execution engines (Temporal, Restate, Inngest) than to agent
libraries; it applies that rigor to multi-agent LLM systems.

| Dimension | CrewAI / AutoGen | LangGraph | Swarm |
|---|---|---|---|
| Routing | LLM-managed (non-deterministic) | Code-defined state machine | Contract-defined, subscription-derived |
| Crash recovery | None | Manual checkpointing | Atomic transitions + event replay |
| Composition | Code refactoring | Python sub-graphs | YAML flow packages with typed pins |
| Agent isolation | Shared chat | Shared state | Scoped sessions, hierarchical addressing |
| Scaling | Single process | Single process | Per-entity parallelism, agent sharding, dynamic instances; runs spanning hours to days |
| Expression language | None | Python | CEL (typed, non-Turing-complete) |
| Audit trail | Logs | Logs | Event store + per-field mutation log |

## The tradeoff, stated plainly

A Swarm flow cannot be re-wired by an LLM at runtime, and the static analyzer will refuse to
boot a half-finished contract. In production that refusal is a feature; during exploration
it is friction. Sketch loose first if you must, then port to Swarm once the workflow
stabilizes.

And one honest limit on the word "deterministic": it stops at the agents. The control loop,
routing, state transitions, and the event log are reproducible, but an LLM is still an LLM, so
re-run a flow and an agent may reason its way to a different answer. What you can always replay
and audit is what the *system* did with that answer, and the fact that the answer only changed
state by passing through a deterministic handler. Swarm does not make the model deterministic; it
makes everything around the model deterministic, which is the part worth auditing, replaying, and
recovering. That is the whole bet, restated as a cost: you accept a fuzzy model and wrap it in a
system that is not.

There is also an upfront price the rest of this page implies but is worth saying outright: real
contracts to write, a Postgres to run, and a model to learn before the first flow boots. The
payoff is everything above; the bill comes first.
