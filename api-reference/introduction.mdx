Swarm exposes one user-facing API: JSON-RPC 2.0 over HTTP for request and response, and
JSON-RPC subscriptions over WebSocket for live streams. One transport, one auth model, one
envelope. The `swarm` CLI, the dashboard, and any custom client all route through it.

This reference is generated from the OpenRPC source of truth (`openrpc.json`), which the spec
generates and treats as authoritative; drift between the two fails boot verification.

<Note>
  Every method is a single `POST /v1/rpc` with the method name in the JSON-RPC body. The
  per-method paths in this section (for example `POST /v1/rpc/run.start`) are a documentation
  convenience so each method gets its own page. See
  [Transport and auth](/api-reference/transport-and-auth) for the real envelope.
</Note>

## Method groups

| Namespace | What it owns |
|---|---|
| `run.*` | Start, control, list, trace, and diagnose runs. |
| `event.*` | Publish, replay, query, and subscribe to events. |
| `entity.*` | List, read, and aggregate entities. |
| `agent.*` | Inspect and control agents; replay deliveries; send directives. |
| `conversation.*` | Sessions, turns, and transcripts. |
| `mailbox.*` | List items and record approve, reject, defer decisions. |
| `runtime.*` | Logs, incidents, pause and resume, destructive reset. |
| `health.*` | Liveness, structured health, and a heartbeat subscription. |
| `rpc.*` | The subscription protocol (`rpc.unsubscribe`). |

## Conventions at a glance

- **Auth:** a bearer token on every HTTP request and the WebSocket upgrade.
- **Pagination:** cursor-based; list methods take `limit` and `cursor` and return
  `next_cursor`.
- **Idempotency:** mutating methods accept an `idempotency_key`.
- **Versioning:** the surface is under `/v1/`; breaking changes bump to `/v2/`.
