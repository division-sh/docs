These patterns combine the [handler fields](/reference/handler-fields) into the shapes that
come up most often. Each is a starting point, not a rule.

## Guard with escalation

Block a handler and route the failure to a named event instead of rejecting it.

```yaml
request.received:
  guard:
    id: rate_limit
    check: "entity.request_count < policy.max_requests_per_hour"
    on_fail: escalate:rate_limit.exceeded
  advances_to: processing
```

`on_fail` is one of `reject` (default), `discard`, `kill`, or `escalate:{event}`.

## Multi-gate pipeline

Track sub-steps with gates, then reset them when re-entering a cycle.

```yaml
review.completed:
  sets_gate: g_reviewed
  advances_to: approved
  emit: approval.requested

resubmission.requested:
  clear_gates: true     # resets every gate on the entity
  advances_to: draft
```

## Accumulate and compute

Wait for several items, aggregate them, then branch on the result.

```yaml
score.received:
  accumulate:
    expected_from: entity.dimensions_requested
    completion: all
    dedup_by: payload.dimension
  compute:
    operation: weighted_average
    keys:
      dimension_key: dimension
      score_keys: [score]
    tiers:
      - dimensions: [quality, speed, cost]
        weight: 0.6
      - dimensions: [innovation, scalability]
        weight: 0.4
    store_as: entity.composite_score
  on_complete:
    - condition: "entity.composite_score >= policy.threshold"
      advances_to: approved
      emit: candidate.approved
    - condition: "entity.composite_score < policy.threshold"
      advances_to: rejected
      emit: candidate.rejected
```

## Fan out

Dispatch one event per item to an agent, then track completion with `accumulate`.

```yaml
batch.requested:
  fan_out:
    items_from: payload.items
    target: worker-agent
    emit:
      event: item.assigned
      broadcast: true
  data_accumulation:
    writes:
      - target_field: expected_count
        expression: "fan_out.count"
```

This is the shape of within-flow [agent sharding](/concepts/agents-and-sessions): each item goes
to an instance of the agent role, and the runtime caps how many run at once (see
[Configuration](/reference/configuration)).

## Rules-based routing

Dispatch on a payload field. `rules` is a list; the matched rule owns the emit.

```yaml
request.received:
  rules:
    - id: billing
      condition: "payload.category == 'billing'"
      advances_to: billing_review
      emit:
        event: billing.requested
        broadcast: true
    - id: technical
      condition: "payload.category == 'technical'"
      advances_to: tech_review
      emit:
        event: tech.requested
        broadcast: true
```

`rules` and `on_complete` are mutually exclusive.

## Dynamic flow instances

Create a per-order or per-customer instance from a template at runtime.

```yaml
order.created:
  action: create_flow_instance
  template: fulfillment
  instance_id_from: payload.order_id
  config_from:
    customer_name: payload.customer_name
    order_total: payload.total
```

Observe all instances with a wildcard subscription such as `fulfillment` then `*` then
`order.completed`.

## Timers

Start a durable timer on a state and cancel it on another.

```yaml
timers:
  - id: payment_deadline
    event: timer.payment_deadline
    delay: "{{payment_deadline_hours}}h"
    start_on: state:awaiting_payment
    cancel_on: state:paid

# handler for the timer event
timer.payment_deadline:
  advances_to: payment_overdue
  emit: payment.overdue_notification
```

## Cross-entity queries

Read across entities to build a report.

```yaml
timer.daily_report:
  query:
    - entities: tickets
      group_by: state
      count: true
      store_as: entity.tickets_by_state
    - entities: tickets
      filter: "priority == 'critical' && state != 'resolved'"
      select: [ticket_id, subject]
      store_as: entity.open_critical
  emit: report.daily_compiled
```

`query` accepts either a single query object or a list of them (as here), and each `store_as`
target is an entity field. `store_as` is always entity-prefixed (`entity.<field>`).

## Payload transform

Build an output payload from entity state, payload, and policy with `emit.fields`.

```yaml
order.finalized:
  emit:
    event: invoice.requested
    fields:
      order_id: entity.order_id
      total: entity.computed_total
      customer: entity.customer_name
```

## Keep reviewers independent

When two agents must not share context, give a role two pools with different subscriptions:
route originals to one and appeals to the other. Same prompt, separate sessions, no shared
context.
