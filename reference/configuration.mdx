Swarm favors CLI flags; each flag has an environment-variable equivalent for deployment.
`swarm verify` is the only command that runs without a runtime or a token. Every other command
targets the runtime's API and needs a bearer token.

## Connecting to the runtime

The runtime exposes one HTTP listener serving health, readiness, `/v1/rpc`, `/v1/ws`, and the
MCP routes. Its default address is **`http://127.0.0.1:8081`**. The bundled Docker Compose
binds that same listener to **`:8070`** instead; aside from the port, the two are the same
surface.

CLI commands resolve connection settings in this order: flag, then environment, then config
file, then built-in default.

<ParamField path="--api-server <url>" type="string">
  Base URL of the runtime. Default `http://127.0.0.1:8081`. Env: `SWARM_API_SERVER`.
</ParamField>
<ParamField path="--api-token-file <path>" type="string">
  File holding the bearer token. Required for every command except `swarm verify`. Env:
  `SWARM_API_TOKEN` (the token value) or `SWARM_API_TOKEN_FILE`. An inline `--api-token` is
  not accepted, to keep secrets out of shell history.
</ParamField>

The optional CLI config file at `$XDG_CONFIG_HOME/swarm/config.yaml` accepts `api_server` and
`api_token_file`.

### swarm run

`swarm run` reaches a runtime in one of two ways:

- **Local (default).** With no `--connect`, it boots a runtime in process on
  `127.0.0.1:8081`. Override the port with `--api-port <n>`.
- **Connected.** With `--connect <url>`, it attaches the run to an already-running runtime
  (started by `swarm serve`) instead of booting its own.

Either way, a bearer token is required.

## Database

Postgres is the store; any Postgres 16 works (local, managed, or a container). The defaults
target a local Postgres, so a stock setup needs no configuration:

<ParamField path="SWARM_DB_HOST / SWARM_DB_PORT / SWARM_DB_NAME / SWARM_DB_USER / SWARM_DB_PASSWORD / SWARM_DB_SSLMODE" type="string">
  Postgres connection parameters. Defaults: `127.0.0.1`, `5432`, `swarm`, `postgres`,
  `postgres`, `disable`. Each `SWARM_DB_*` variable falls back to the standard libpq
  equivalent (`PGHOST`, `PGPORT`, `PGDATABASE`, `PGUSER`, `PGPASSWORD`) before the default.
</ParamField>

## Runtime mode and credentials

<ParamField path="SWARM_LLM_RUNTIME_MODE" type="string">
  The agent runtime: `api` or `cli_test`. The runtime validates its credential at boot, even
  if the flow you run uses no agents.
</ParamField>

- **`cli_test`** drives the Claude CLI as a subprocess. Requires `CLAUDE_CODE_OAUTH_TOKEN` and
  the tool-gateway variables `SWARM_TOOL_GATEWAY_URL`, `SWARM_TOOL_GATEWAY_CONTAINER_URL`, and
  `SWARM_TOOL_GATEWAY_TOKEN` (the repository `.env` provides the gateway variables).
- **`api`** uses the Anthropic API. Requires `ANTHROPIC_API_KEY`.

Credentials are read from the environment and a local credential store, and are never logged
or persisted.

## Storage

<ParamField path="SWARM_ARTIFACT_ROOT" type="string">
  Runtime-private root for git repositories written by `artifact_repo_commit`. Default
  `/var/lib/swarm/artifacts`. Never mounted into agent sessions.
</ParamField>

## Budget caps

The `budget_*_percent` thresholds in `policy.yaml` are percentages of a monthly USD cap, set in
the `budget:` configuration section. A scope is monitored only when its cap is above zero. See
[Budget and cost](/operate/budget-and-cost).

```yaml
budget:
  global_monthly_cap: 500       # USD per month across the deployment
  per_entity_monthly_cap: 5     # USD per month per entity
  system_monthly_cap: 100       # USD per month for platform/system work
```
