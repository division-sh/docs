Swarm runs as a single Go binary backed by Postgres. Build the binary and run it against a
Postgres container, or use Docker Compose for a fully containerized runtime.

## Requirements

- **Docker Compose path:** Docker and Docker Compose.
- **From-source path:** Go 1.23, Postgres 16.

## Install paths

<Tabs>
  <Tab title="Docker Compose">
    The repository ships a `docker-compose.yml` with a `postgres` service and an
    `orchestrator` service.

    ```bash
    git clone https://github.com/division-sh/swarm && cd swarm
    export SWARM_CONTRACTS_HOST_DIR=/absolute/path/to/your/contracts
    docker compose up -d postgres orchestrator
    ```

    - The orchestrator binds its HTTP listener to **`:8070`** here. The runtime's default is
      `:8081`; Compose overrides the port. It serves health, readiness, and the
      `/v1/rpc` + `/v1/ws` API on that one listener.
    - `SWARM_CONTRACTS_HOST_DIR` is mounted read-only at `/opt/swarm/contracts`.
    - The Compose orchestrator runs the `cli_test` backend (drives the Claude CLI, needs
      `CLAUDE_CODE_OAUTH_TOKEN`). Outside Compose, select a backend with `SWARM_LLM_BACKEND`
      (`api`, `cli_test`, or `openai_compatible`); the retired `SWARM_LLM_RUNTIME_MODE` is rejected.
  </Tab>
  <Tab title="Build from source">
    ```bash
    go build ./cmd/swarm
    ```

    Start Postgres (any 16; a container matching the repo's `.env` works), then start the
    runtime:

    ```bash
    docker run -d --name swarm-postgres \
      -e POSTGRES_PASSWORD=postgres -e POSTGRES_DB=swarm \
      -p 5432:5432 postgres:16

    swarm serve --contracts ./my-flow
    ```

    `swarm serve` is the long-running runtime, the API, health, and MCP startup owner, reachable
    at **`http://127.0.0.1:8081`** by default. (The [Quickstart](/quickstart)'s `swarm run` is the
    one-shot, in-process alternative; `serve` is what you operate.) It needs Postgres, an API
    token (`SWARM_API_TOKEN`), and an LLM credential (see below). CLI commands target that same
    address; point them elsewhere with `--api-server`. See
    [Configuration](/reference/configuration).
  </Tab>
</Tabs>

## Postgres

Postgres is the substrate, not an add-on. Every event lands in the event store, every entity
state change lands in a separate mutation log, and runs are correlated by `run_id`, which is
what makes replay and fork possible.

Any Postgres 16 works: a local install, a managed service, or a container. Docker is not
required. The binary defaults to a local Postgres (`127.0.0.1:5432`, database `swarm`, user
and password `postgres`) and reads `SWARM_DB_*` (falling back to the standard `PG*`
variables) for anything else. Create the `swarm` database, or set `SWARM_DB_NAME`.

## Credentials and runtime mode

Beyond `swarm verify`, you always need an API token, and an LLM credential once your flow runs
agents:

- **An API token.** Set `SWARM_API_TOKEN` (any value for local development). Every command
  except `swarm verify` requires it.
- **An LLM credential**, needed only when your flow has agents; an agent-free flow boots without
  one. Select a backend with `SWARM_LLM_BACKEND`: `cli_test` (drives the Claude CLI; needs
  `CLAUDE_CODE_OAUTH_TOKEN` plus the tool-gateway variables that `.env` provides), validated at boot
  when the bundle declares agents; `api` (the Anthropic API; needs `ANTHROPIC_API_KEY`), checked at
  the first agent turn; or `openai_compatible` (any Chat Completions endpoint; needs
  `OPENAI_COMPATIBLE_API_KEY`, a base URL, and a default model).

Credentials are read from the environment and a local credential store, and are never logged
or persisted. See [Configuration](/reference/configuration) for the full list.

## Verifying the install

```bash
# Validate a contract bundle without a running runtime (local, no DB needed)
swarm verify --contracts ./my-flow

# Print binary version; add --server to query a running runtime
swarm version
```

`swarm verify` is the one command that works purely against files on disk, every other
command targets the running runtime's API.
