Two nouns sit at the center of every Swarm system, and it helps to separate them up front. A
**flow** is the definition: a self-contained package of YAML contracts (states, events, system
nodes, agents, tools, policy) that describes a piece of work. An **entity** is one actual unit
of that work moving through it: a ticket, an order, a claim. If a flow is the program, an entity
is the record it processes, advancing through the flow's states as events arrive.

A flow is also the only building block. Every level of a Swarm system is a flow, from the root
down to the smallest child; there is no separate "project." And a flow exposes a typed public
interface (its **pins**), which is what lets you drop one into another system and wire it
without refactoring.

## Instances: static vs template

A flow is a definition; a **flow instance** is one running copy. How many copies exist depends
on the mode you declare in the parent:

| Mode | Instances | Created | Use it for |
|---|---|---|---|
| `static` | exactly one | at boot | most flows |
| `template` | zero at boot, many at runtime | a `create_flow_instance` action | one copy per customer, per order, and so on |

A static flow is the common case: one copy boots, and many entities flow through it (a single
ticket flow handles thousands of tickets). A template flow is a stamp: nothing runs at boot, and
the runtime mints a fresh instance whenever it needs one, for example one order flow per order.

## Entities

An entity is the thing moving through a flow instance's state machine. It carries three things:

- **state**: which named state it is in (`new`, `assigned`, `resolved`), exactly one at a time.
- **gates**: named boolean checkpoints a handler can set and a guard can check (`verified: true`).
- **fields**: typed data that handlers write (`category`, `total`, `resolution`).

You declare an entity's fields in the flow's `entities.yaml`. Its state is not declared there,
because state belongs to the flow's state machine (in `schema.yaml`).

```yaml entities.yaml
order:
  status: text
  total: numeric
  placed_at:
    type: timestamp
    immutable: true        # set once, at creation
```

## Many entities, no locks

Many entities live in one flow at the same time, each with its own id and its own state. You
never reason about locking: the engine processes the events for a single entity one at a time,
so that entity's state and data cannot be corrupted by a race, while events for *different*
entities run concurrently. That is what lets one flow carry hundreds or thousands of entities in
parallel without any author-managed concurrency.

This multiplicity has a knock-on effect: because a flow holds many entities at once, an event
arriving from another flow cannot just go "to the flow," it has to say which entity it is for.
That is the core of why cross-flow wiring needs targets; see [Composing flows](/build/composition).

## Pins: a flow's public interface

A flow keeps most of its events and data private. What it chooses to expose, it exposes through
typed **pins**: the events it accepts from outside, the events it emits outward, and the entity
fields it reads or writes (no two flows may write the same field). Pins are the contract other
flows wire against, which is what makes composition mechanical rather than a refactor. Any event
not exposed as a pin is internal to the flow.

See [Composing flows](/build/composition) for how pins are wired across flows.
